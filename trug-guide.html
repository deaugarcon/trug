trug

Your iPhone's data. Your Mac. Nowhere else.

trug is a free, open-source command-line tool for macOS that backs up your iPhone over USB and lets you read what's inside — your messages, contacts, call history, and notes — as files you actually own. No cloud, no account, no telemetry. Everything happens on your Mac.

# the whole idea, in four commands
$ trug devices list          # see your connected iPhone
$ trug backup create <udid>  # back it up, locally
$ trug backup inspect <udid> messages   # redacted preview
$ trug backup export <udid> notes --out my-notes.json
What it is

A backup tool that answers to you

Your iPhone can already back itself up to a Mac — but the result is a folder of hashed files you can't read, made by tools you can't script. trug creates those same device backups and gives you a window into them: a safe, redacted preview for browsing, and a full export when you decide you want your data as ordinary JSON or CSV.

  • Local-only. trug talks to your phone over a USB cable and to your disk. It has no server, sends nothing anywhere, and collects nothing about you.
  • Reads your own backups. Encrypted backups stay encrypted at rest; trug unlocks them only in memory, only when you provide your backup password.
  • Scriptable. It's a real CLI with stable exit codes and machine-readable output — cron it, pipe it, build on it.

Current status: alpha (v0.1.1-alpha). Backup creation, verification, browsing, file extraction, encryption management, and inspect/export of four data stores are in place. Restore is not yet supported — your existing Finder/iTunes restore path is untouched.

Install

One line with Homebrew

$ brew install deaugarcon/tap/trug
$ trug --version
0.1.1-alpha

Runs on macOS 14 (Sonoma) or later, Apple silicon and Intel. Building from source instead? See Build (dev) in the README — you'll need Xcode 16+ and Homebrew.

macOS 27 beta: the Command Line Tools 27 beta ships a SwiftPM defect that breaks source builds. Install full Xcode 27 beta and select it with sudo xcode-select -s /Applications/Xcode-beta.app/Contents/Developer first.
Quick start

First backup to first export

  1. Plug in your iPhone and find it

    Connect over USB (unlock the phone and tap "Trust" if asked), then:

    $ trug devices list

    You'll see your device's name and its UDID — the long identifier you'll pass to every backup command. Add --json if you're scripting.

  2. Create a backup

    $ trug backup create <udid>

    The backup lands in trug's local store on your Mac. If your device has backup encryption enabled (recommended — it's what makes saved passwords and health data part of the backup), trug will need your backup password to read it later, never to create it.

  3. Peek inside — safely

    $ trug backup inspect <udid> messages

    inspect is read-only and deliberately blunted: rows are capped, names and numbers are masked, text is truncated. It exists so you can confirm "yes, this backup has what I think it has" without your data ever landing on screen in full.

  4. Export for real

    $ trug backup export <udid> notes --out my-notes.json
$ trug backup export <udid> contacts --out contacts.csv --format csv

    export writes the complete, unmasked store to a file you name. It's the only command that writes your data to disk, it creates the file owner-read-only (0600), and it refuses to overwrite an existing file unless you pass --force.

Your backup password, when asked

For encrypted backups, trug asks for the backup password with a silent prompt (nothing is echoed). In scripts, set it via the environment instead — ideally straight from your keychain so it never appears in your shell history:

$ TRUG_BACKUP_PASSWORD=$(security find-generic-password -s my-backup -w) \
    trug backup export <udid> messages --out messages.json
Reference

The command surface

CommandWhat it does
trug devices listShow connected iPhones with name and UDID. (Real identifiers, from your own attached hardware.)
trug backup createCreate a local backup of a connected device.
trug backup listList backups in trug's local store.
trug backup verifyCheck an existing backup's integrity.
trug backup browseList the files inside a backup, optionally one domain.
trug backup extractPull one file out of a backup by path.
trug backup inspectRedacted, capped, read-only preview of one store: messages, contacts, calls, or notes.
trug backup exportFull, unmasked export of one store as JSON (default) or CSV, to a file you name with --out.
trug backup encryptionTurn device backup encryption on, or rotate the password (TRUG_BACKUP_NEW_PASSWORD for scripting).

Every subcommand has full help: trug help backup export.

The core idea

Inspect shows shapes. Export shows everything.

Masked sender, masked number, text truncated at 40 characters, row count capped at the database layer. Nothing to shoulder-surf.

from
Ada Lovelace
number
+1 555 010 5542
date
2026-06-14 09:31
text
Landing at 4 — can you grab the analytical engine from the shop?

The complete record, in a 0600 file, at a path you chose, only because you asked. (Example data is synthetic.)

Notes get their full body — with two honest exceptions

Since v0.1.1, export notes includes each note's complete decoded text (schema_version: 2). Two rules keep it honest:

  • A locked (password-protected) note's body is withheld as an empty string. trug never attempts to decode locked content.
  • A body trug cannot decode is omitted — never guessed, and never the reason an export fails. You still get the note's title and metadata.

CSV that won't bite back

CSV exports are RFC-4180 quoted, and any field starting with =, +, -, @, or a tab/CR/LF is neutralized so a message that says "=SUM(…)" opens in your spreadsheet as text, not as a formula.

Privacy model

Four rules the code is built around

P1

No decrypted working file survives. Anything trug decrypts to read goes into an owner-only temp file that is removed when the command exits — normally or on error.

P2

Every run starts with a sweep. Before touching your data, trug scrubs any stale working files a previous crash might have left behind.

P3

The preview is blunted at the source. inspect's caps and masks are applied in the database query itself — full rows never reach the display code.

P4

One door for output. export is the only path that writes your data to disk: explicit --out, owner-read-only permissions, no silent overwrites, no symlink tricks.

Your backup password is never stored by trug and never appears in any file it writes. Full details in SECURITY.md.

Good to know

Current limits, plainly

  • No restore yet. trug creates and reads backups; restoring one to a phone is planned, not shipped.
  • USB is the supported path. Wi-Fi device info is experimental.
  • Validated on iOS 27. Older iOS versions (e.g. iOS 18) should work but carry a caveat until device-verified.
  • Four stores today. Messages, contacts, call history, notes. Photos and other domains: backup browse + backup extract can pull raw files in the meantime.
  • Alpha means alpha. Interfaces may still move; the changelog and releases page tell the truth: releases.
